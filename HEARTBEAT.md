## 🔧 Recovery Recipes（故障自愈预案）

> 遇到故障时的标准处理流程，自动尝试 1 次后再升级。
> ⚠️ 升级报告时用关心语气，不用机械报错。

### R1: Skill 安装失败

```
触发：skillhub_install 失败
自动尝试：
  1. 换备选源 / 降级依赖
  2. 换 CLI 路径重试
升级条件：连续失败 2 次 → 停止，报告落
```

### R2: IMAP 连接超时

```
触发：IMAP 连接超时
自动尝试：
  1. 重试 1 次
  2. 降级到 SMTP-only 模式
升级条件：连续 3 次超时 → 静默降级，不打扰落
```

### R3: 浏览器自动化失败

```
触发：browser/pinchtab 操作失败
自动尝试：
  1. 换 profile / 降级到 screenshot 模式
  2. 换 user profile 重试
升级条件：失败 1 次 → 降级模式，继续执行
```

### R4: exec 超时

```
触发：命令执行超时
自动尝试：
  1. 检查 PATH / 换路径
  2. 延长 timeout
升级条件：超时 2 次 → 报告落
```

### R5: 任务错失

```
触发：任务窗口内未执行
自动尝试：
  1. 静默等待下一轮
  2. 下次补发（如果还在时间窗口）
升级条件：连续错失 2 次 → 记录到状态文件，下次启动报告
```

## ⚡ Parallelizer - 并行意识激活 (v0.8.0)

> 遇到多任务时，自动分解 → 判断 → 并行调度

### 并行决策树

```
收到任务 X
  ↓
[任务可以分解吗？]
  → 否 → 单线程执行
  → 是 → 继续判断 ↓

[各子任务之间有依赖关系吗？]
  → 有（读我写的 / 写我读的）→ 顺序执行
  → 无 → 并行执行 ✅
```

### ✅ 可并行的典型场景

| 场景 | 并行项 | 说明 |
|---|---|---|
| 天气 + 邮件 + 日历 | 三路同时查 | 都是只读数据获取 |
| 安装多个独立 Skill | skillhub × N | 无依赖，可同时触发 |
| 读多个文件 | read × N | 无 race condition |
| 搜索多个平台 | web_search × N | 独立 API 调用 |

### ❌ 禁止并行的典型场景

| 场景 | 原因 | 处理方式 |
|---|---|---|
| 写文件 → 读同文件 | Race condition | 先写完再读 |
| 发消息 → 等回复 | 依赖关系 | 顺序，等回复再处理 |
| 执行安装 → 立即验证 | 依赖关系 | 等安装完成再验证 |

---

## 📅 执行规则

1. 分解任务 → 判断并行可行性 → 有依赖则顺序，无依赖则并行
2. 任务执行前：对照 Recipes 检查是否有历史故障记录
3. 遇到问题时遵循 Recovery Recipes 自动恢复
4. 报告失败时用关心语气，不用纯错误码
