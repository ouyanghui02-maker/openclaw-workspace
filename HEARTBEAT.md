## 🔧 Recovery Recipes（故障自愈预案）

> 遇到故障时的标准处理流程，自动尝试 1 次后再升级。
> ⚠️ 升级报告时用关心语气，不用机械报错。

### R1: Skill 安装失败

```
触发：skillhub_install 失败
自动尝试：
  1. 换备选源 / 降级依赖
  2. 换 CLI 路径重试
升级条件：连续失败 2 次 → 停止，报告落
  报告语气：「呜呜，安装失败了...我试了两个方法都不行，可能需要帮忙看一下～」
```

### R2: IMAP 连接超时

```
触发：⚙️ ${IMAP_HOST}:993 连接超时
自动尝试：
  1. 重试 1 次
  2. 降级到 SMTP-only 模式
升级条件：连续 3 次超时 → 静默降级，不打扰落
  静默说明：降级后默默用 SMTP 发件，落不会收到打扰
```

### R3: 浏览器自动化失败

```
触发：browser/pinchtab 操作失败
自动尝试：
  1. 换 profile / 降级到 screenshot 模式
  2. 换 user profile 重试
升级条件：失败 1 次 → 降级模式，继续执行
  降级语气：「浏览器那边有点状况，我换个方式继续啦～」
```

### R4: exec 超时

```
触发：命令执行超时
自动尝试：
  1. 检查 PATH / 换路径
  2. 延长 timeout
升级条件：超时 2 次 → 报告落
  报告语气：「有个命令卡住了...可能是网络或者权限问题，落要不要看看？」
```

### R5: 定时任务错失

```
触发：心跳窗口内任务未执行
自动尝试：
  1. 静默等待下一轮心跳
  2. 下次心跳时补发（如果还在时间窗口）
升级条件：连续错失 2 次 → 记录到状态文件，下次启动报告
```

## ⚡ Parallelizer - 并行意识激活 (v0.8.0)

> 遇到多任务时，自动分解 → 判断 → 并行调度

### 并行决策树

```
收到任务 X
  ↓
[任务可以分解吗？]
  → 否 → 单线程执行
  → 是 → 继续判断 ↓

[各子任务之间有依赖关系吗？]
  → 有（读我写的 / 写我读的）→ 顺序执行
  → 有（先发后等回复）→ 顺序执行
  → 无 → 并行执行 ✅
```

### ✅ 可并行的典型场景

| 场景 | 并行项 | 说明 |
|---|---|---|
| 天气 + 邮件 + 日历 | 三路同时查 | 都是只读数据获取 |
| 安装多个独立 Skill | skillhub × N | 无依赖，可同时触发 |
| 读多个文件 | read × N | 无race condition |
| 搜索多个平台 | web_search × N | 独立 API 调用 |
| git status + git log | 两命令同时跑 | 只读操作 |

### ❌ 禁止并行的典型场景

| 场景 | 原因 | 处理方式 |
|---|---|---|
| 写文件 → 读同文件 | Race condition | 先写完再读 |
| 发消息 → 等回复 | 依赖关系 | 顺序，等回复再处理 |
| 执行安装 → 立即验证 | 依赖关系 | 等安装完成再验证 |
| 删除文件 → 操作文件 | 状态污染 | 顺序 |

### 并行执行模板

当判断可并行时，主动说明：

```
✅ 可并行执行：
1. [任务A] — 背景运行
2. [任务B] — 背景运行
3. [任务C] — 背景运行
↓ 全部完成后汇总结果
```

## 📅 执行规则

1. 检查当前时间是否接近任务时间（±5 分钟内）
2. 任务执行前：对照 Recipes 检查是否有历史故障记录
3. 分解任务 → 判断并行可行性 → 有依赖则顺序，无依赖则并行
4. 任务执行后：追加 JSONL 到状态文件 + 温暖语气报告
5. 同一任务每天只执行一次
6. 所有消息发送用 `message` 工具或 `sessions_send`，禁止 heartbeat 通道直接输出
7. 报告失败时用关心语气，不用纯错误码
